# PA Assistant — Comprehensive Improvement & Migration Document

**Version:** 1.2 | **Date:** 2026-03-19
**Audience:** Alexey + open-source contributors
**Goal:** Convert `PersonalWorkTrackerApp` (Python CLI) → a Google Drive Add-on, publishable on Google Workspace Marketplace, installable by non-technical users in one click.

***

## Part 1 — Critical Issues to Fix Before Anything Else

These are blockers that affect correctness and security regardless of the migration path.

### 1.1 Replace Service Account Auth with User OAuth 2.0 (CRITICAL)

The current service account model (`pa_assistant/connectors/drive.py:L41`) is high-friction for a personal tool. A service account cannot see a user's personal Drive unless every file is explicitly shared with it, or Domain-Wide Delegation is configured.

**Action:** Replace `google.oauth2.service_account.Credentials` with a standard OAuth 2.0 user flow using `google_auth_oauthlib.flow.InstalledAppFlow`. This will allow the app to act on behalf of the actual user. Store the resulting `token.json` locally (and ensure it is gitignored). Use a `setup_auth.py` script to automate this for team members.

### 1.2 Switch from `files.list` to Drive Activity API v2 (CRITICAL)

Currently, `pa_assistant/connectors/drive.py:L64` queries `drive_service.files().list()` filtered by `modifiedTime`. This misses: file opens/reads, comments, renames, moves, and shares. The Drive Activity API v2 is the correct "source of truth" for work signals.

**Action:** Update `pa_assistant/connectors/drive.py` (or create a new `drive_activity.py` connector) to use the `driveactivity` service. Request scope `https://www.googleapis.com/auth/drive.activity.readonly`.

### 1.3 Stabilize Configuration & Log Paths (HIGH)

While `pa_config.yaml` exists, some values are still hardcoded or inconsistently handled:

- `llm_logs.py:L45` and `L88` hardcode paths to `~/.personal-summarizer/`. These should be configurable via `pa_config.yaml`.
- Ensure `pa_config.yaml` is the single source of truth for `reports_dir` (it is correctly fetched in `runner.py:L50`, but default offsets like `default_start_offset_days` seen in the YAML aren't fully utilized in the current `cli.py`).

**Action:** Refactor `pa_assistant/runner.py` to pass a full `RunnerConfig` object to connectors, ensuring all paths (including LLM logs) are resolved from the YAML configuration.

## Part 2 — Architecture Decision: Add-on Strategy

The expert team recommends a **hybrid architecture** for the final product:

| Layer | Technology | Rationale |
| :-- | :-- | :-- |
| UI / Sidebar | Google Apps Script + HTML Service | Native Drive integration, runs in browser, zero install friction. |
| Drive Activity Fetch | Apps Script Drive Activity Advanced Service | Handles OAuth automatically; native access to user events. |
| Logic & Processing | Apps Script (JavaScript) | Port the normalization and sessionization logic to JS. |
| LLM Summarization | Python (Cloud Run/Functions) | If complex AI summaries are added, keep the Python layer as an API. |
| Report Storage | Google Drive (Docs) | Save reports directly as Google Docs instead of local MD files. |

## Part 4 — Data Model & Processing Improvements

### 4.1 Implementation of Sessionization

The current tool provides a flat chronological table (`report.py:L48-57`). To make it useful for "picking up where you left off," we must group activities.

**Algorithm:**

- Use a `SESSION_GAP` (e.g., 30 mins).
- If `activity[i].timestamp - activity[i-1].timestamp > SESSION_GAP`, start a new group.
- **Benefit:** "Edited 5 files" becomes "Morning Session: Worked on Project X for 2 hours."

### 4.2 Enriching the `ActivityEntry` Model

Current `models.py` captures basic metadata. We should expand `ActivityEntry` to include:

- `action_type`: (VIEW, EDIT, RENAME, etc.) from the Activity API.
- `session_id`: To support the new grouped UI.

## Addendum A — Report Format & Output Design

### A.1 Report Structure: From Flat Table to Narrative Timeline

The current flat Markdown table in `report.py:L48-57` must be replaced with a two-level hierarchy: a **summary header block** followed by **grouped session blocks**. This is the single highest-impact change for the tool's core value proposition.

#### A.1.1 Summary Header Block

Every report must open with a TL;DR block answering "what was I doing?" in under 3 seconds:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 PA REPORT · Mar 13–19, 2026
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⏱  Total active time:   6h 12min across 5 sessions
📁 Files touched:        23 (14 edited, 6 viewed, 3 created)
🔥 Most active day:      Tuesday Mar 17 (3 sessions)
📌 Focus filter:         "VeriQ onboarding"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

All fields in this block are derived from the sessionized activity data — no LLM required.

#### A.1.2 Session Block Format

```
🟦 SESSION 1 · Monday Mar 16 · 09:12–11:05 · 1h 53min · 4 files · Mostly editing
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ✏️  EDIT    · VeriXQ_PRD_v3.docx           → [Open ↗]
  ✏️  EDIT    · Architecture_Diagram.drawio  → [Open ↗]
  💬  COMMENT · API_Spec_Draft.docx          → [Open ↗]
  📄  CREATE  · Meeting_Notes_Mar16.docx     → [Open ↗]
```

Each session header contains:

| Field | Source |
|---|---|
| Session number | Sequential counter from `processor.py` |
| Day + date | Derived from first event timestamp |
| Time range | First and last `ActivityEntry.timestamp` in session |
| Duration | Calculated from time range |
| File count | Count of unique files in session |
| Dominant action | Most frequent `action_type` in session |

#### A.1.3 Action Type Icon Vocabulary

Use this consistent mapping for all output formats (Markdown, Google Doc, sidebar HTML):

| `action_type` | Icon | Label |
|---|---|---|
| `EDIT` | ✏️ | EDIT |
| `CREATE` | 📄 | CREATE |
| `VIEW` | 👁️ | VIEW |
| `COMMENT` | 💬 | COMMENT |
| `RENAME` | 🏷️ | RENAME |
| `MOVE` | 📦 | MOVE |
| `DELETE` | 🗑️ | DELETE |

These map directly to Drive Activity API v2 action types introduced in Phase 0.

***

### A.2 "Resume Here" Section

Append a final section to every report, auto-generated from activity data alone (no AI needed):

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🚀 RESUME HERE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Last file edited:    VeriQ_PRD_v3.docx   [Open ↗]  · Tue Mar 17, 15:42
Last file created:   Sprint_Plan_v2.docx [Open ↗]  · Tue Mar 17, 14:10
Last session ended:  Tuesday Mar 17 at 15:42
```

**Implementation:** Derive from `max(timestamp)` per `action_type` across all sessions. Add this as the final method in `report.py` (or the Jinja2 template introduced in Phase 1).

***

### A.3 Report Naming Convention

Use ISO week-number naming for all saved report files. This makes reports sortable and findable without remembering exact dates:

```
PA_Report_2026-W12.md          ← weekly (recommended default)
PA_Report_2026-03-13_to_2026-03-19.md   ← date range (fallback for custom periods)
```

Update the `reports_dir` output logic in `runner.py` accordingly.

***

### A.4 `--period` Shorthand CLI Arguments

Add human-friendly period shortcuts to `cli.py` to replace the mandatory `--start-date` / `--end-date` pair. These should map internally to `start_date` / `end_date` before being passed to `runner.py`:

| Shorthand | Resolves to |
|---|---|
| `--period today` | today 00:00 → now |
| `--period yesterday` | yesterday 00:00 → 23:59 |
| `--period 7d` | now minus 7 days → now |
| `--period this-week` | Monday 00:00 of current week → now |
| `--period last-week` | Monday 00:00 → Sunday 23:59 of previous week |

The existing `default_start_offset_days` key in `pa_config.yaml` (currently not fully utilized in `cli.py`) should be used as the default when no `--period` or `--start-date` is provided.

**Estimated effort:** 30 minutes in `cli.py`.

## Addendum B — Output Format Strategy (CLI vs. Add-on)

### B.1 Design Principle

The output format must match the context in which it is consumed:

| Context | Primary Output | Rationale |
|---|---|---|
| CLI (current, your team) | Markdown file | Fast, portable, renders in Obsidian and any Markdown viewer |
| Add-on (Phase 2) | Google Doc + linked Google Sheet | Narrative-first for non-technical users, data depth one click away |

### B.2 Recommended Add-on Output: Google Doc with Linked Sheet (Option E)

After evaluating five output strategies, the recommended approach for the add-on is a **Google Doc as the primary output, with a companion Google Sheet linked inside it**.

**How it works:**

1. The tool generates a Google Sheet with two tabs:
   - **Tab 1 — Sessions:** One row per session with summary fields (date, duration, file count, dominant action)
   - **Tab 2 — Raw Activity:** One row per `ActivityEntry` (timestamp, file name, action type, link, session ID)
2. The tool generates a Google Doc containing:
   - The summary header block (§A.1.1)
   - The "Resume Here" section (§A.2)
   - The session timeline (§A.1.2)
   - An **embedded linked table** pulled from Sheet Tab 1, allowing the user to open the Sheet for full filtering/sorting with one click

**Why this over alternatives:**

| Option | Verdict | Reason |
|---|---|---|
| Doc only | ❌ No data depth | Can't filter/sort; loses audit trail |
| Sheet only | ❌ Cold UX | Doesn't answer "where was I?" naturally |
| Doc + Sheet (separate files) | ❌ Drive clutter | Two unlinked files per period; confusing |
| Sheet with dashboard tab | ⚠️ Acceptable fallback | Fiddly to style; still feels like a spreadsheet |
| **Doc with linked Sheet (recommended)** | ✅ Best of both | Warm narrative default + full data power one click away |

**Implementation note for Phase 2:** Create the Sheet first (Google Sheets API), then create the Doc (Google Docs API) and insert the linked range using `sheets.spreadsheets.values` embedded via `docs.documents.batchUpdate`. Store both file IDs in `PropertiesService.getUserProperties()` so the add-on can reuse and update them on subsequent runs rather than creating new files each time.

### B.3 Session Gap — User-Configurable Setting

The `SESSION_GAP` threshold (default: 30 minutes) must be exposed as a visible setting — not buried in YAML for the CLI or hidden in code for the add-on.

**CLI:** Add `session_gap_minutes` key to `pa_config.yaml` with a default of `30` and a comment explaining its effect.

**Add-on sidebar:** Render as a simple dropdown in the settings panel:

```
Group activities into sessions when gap exceeds:
  ○ 15 minutes   ● 30 minutes   ○ 60 minutes   ○ 2 hours
```

This single control has an outsized effect on how sessions are shaped and users will have strong intuitions about their own work rhythm — expose it early.
